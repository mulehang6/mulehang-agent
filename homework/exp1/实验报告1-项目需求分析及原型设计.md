# 软件项目管理实验报告一

## 一、实验名称

项目需求分析及原型设计

## 二、实验目的

1. 结合真实项目，理解需求分析、信息架构、线框设计和原型说明在软件项目管理中的作用。
2. 学会从现有技术方案中抽取用户需求、核心流程和功能模块，形成可用于沟通的产品描述。
3. 通过文本化原型设计方法，完成项目早期界面与交互方案表达，为后续实现提供依据。
4. 训练从“技术架构文档”向“用户可感知产品形态”转换的能力，提升需求建模与设计表达能力。

## 三、实验环境

- 操作系统：Windows 11
- 开发项目：`mulehang-agent`
- 项目语言：Kotlin/JVM
- 构建工具：Gradle
- 文档形式：Markdown
- 原型表达方式：文本化页面原型 + 信息架构图 + 交互流程说明

## 四、实验选题说明

本次实验选题为本人当前正在进行的软件项目 `mulehang-agent`。该项目是一个基于 Kotlin/JVM 与 JetBrains Koog 的 agent 实验与重建仓库，当前总体路线采用 `runtime-first` 架构，优先建设 runtime、BYOK、agent 集成与 CLI 主入口，再逐步扩展 ACP 和未来可选的 Web UI / desktop client surface。

由于实验报告本次允许采用纯文本形式提交，因此本报告不强行制作完整 Axure 原型，而是基于现有项目文档与开发计划，输出一份面向未来界面的文本化需求分析与原型设计说明。该说明的目标不是直接交付正式产品，而是完成实验一所要求的需求梳理、信息架构设计、页面原型表达和交互流程说明。

## 五、实验理论基础

### 1. 需求分析

需求分析的核心是识别目标用户、用户场景和系统价值。在项目早期，需求分析决定了系统边界，也直接影响后续的功能拆分、优先级排序与项目计划安排。

### 2. 信息架构

信息架构用于描述系统中的页面组织关系、功能层级和导航方式。清晰的信息架构可以降低后续设计和开发过程中的沟通成本。

### 3. 线框图与原型

线框图强调页面结构、模块布局和信息优先级；原型则进一步表达交互逻辑、状态切换和用户操作结果。课程实验中的原型重点在于“可演示”和“可说明”，不等同于完整业务系统。

### 4. 文本化原型的适用性

当项目处于架构设计期、正式界面尚未实施时，可以采用 Markdown、流程图、页面说明和交互步骤等方式表达原型思想。该方法虽然不具备高保真视觉效果，但依然可以完成需求确认、页面规划和交互梳理。

## 六、项目背景与需求分析

### 1. 项目核心价值

`mulehang-agent` 的核心价值可以概括为：

> 为开发者提供一个可扩展、可配置、以 runtime 为中心的 agent 运行框架，使模型接入、工具能力接入和不同客户端表层能够在统一契约下协同工作。

### 2. 目标用户画像

| 用户角色 | 角色描述 | 核心需求 | 典型场景 |
| --- | --- | --- | --- |
| 个人开发者 | 需要快速接入大模型并本地验证 agent 能力的开发者 | 配置 provider、查看模型、发起会话、观察结果流 | 本地调试新模型或验证工具调用流程 |
| 项目维护者 | 负责维护 agent 框架演进与能力边界的仓库维护者 | 明确 runtime、provider、capability、client surface 之间的边界 | 为后续 CLI、ACP、Web UI 制定统一扩展方式 |
| 集成使用者 | 未来希望将该 agent 能力接到其他客户端或管理界面的使用者 | 直观看到配置状态、调用状态、错误信息和结果输出 | 在可视化界面中配置模型并执行任务 |

### 3. 主要问题与需求

结合仓库现有文档，可以归纳出以下主要需求：

1. 需要统一管理 `custom provider`、`baseUrl`、`apiKey`、提供商类型与模型发现结果。
2. 需要通过统一 runtime 发起请求，避免不同入口各自维护独立调用流。
3. 需要对工具能力、MCP 能力和 direct HTTP internal API 做统一接入。
4. 需要在用户侧体现会话状态、流式输出、错误提示和结构化结果。
5. 未来如果扩展 Web UI，也必须复用现有 runtime 与 agent 主线，而不是重写一套系统。

## 七、核心功能模块分析

根据项目设计文档，系统可以抽象为以下功能模块：

| 模块名称 | 主要职责 |
| --- | --- |
| Provider 配置模块 | 新增、编辑、保存 `custom provider`，配置 `baseUrl`、`apiKey` 和提供商类型 |
| 模型发现模块 | 在连接成功后自动探测远端模型列表，并形成可选 binding |
| Runtime 会话模块 | 创建 session、维护 request context、组织事件流和错误边界 |
| Agent 执行模块 | 基于 Koog `AIAgent` 执行请求，处理策略、输出和能力调用 |
| Capability 集成模块 | 统一接入 local/custom tools、MCP、direct HTTP internal API |
| CLI 展示模块 | 负责用户输入、流式结果显示、状态提示和错误展示 |
| 输出与入口适配模块 | 负责把 runtime 的统一输出映射到具体入口，并保证后续扩展时不破坏主轴 |

## 八、信息架构设计

本项目的信息架构不按 Web 页面层级展开，而是按运行时核心功能展开。这样更符合仓库当前 `runtime-first` 的设计主线，也更能体现系统请求从进入、解析、执行到输出的完整流动方式。

```text
mulehang-agent Runtime Core
├── 输入入口层
│   ├── CLI 请求输入
│   └── 后续可扩展入口适配
├── Runtime 主轴
│   ├── Session
│   ├── Request Context
│   ├── Capability Request
│   └── Error Boundary
├── Provider 与 Binding 层
│   ├── Custom Provider Profile
│   ├── 提供商类型
│   ├── Connection Probe Result
│   ├── Model Discovery Result
│   └── Executor / Model Binding
├── Agent 与 Capability Integration 层
│   ├── Koog AIAgent
│   ├── Local / Custom Tools
│   ├── MCP
│   └── Direct HTTP Internal API
└── 输出层
    ├── RuntimeEvent
    ├── RuntimeResult
    └── RuntimeFailure
```

该信息架构的设计原则是：

1. 输入、执行、能力接入和输出四类职责相互分离。
2. 保持和仓库文档中的 runtime-first 主线一致。
3. 任何入口都只能消费统一 runtime 契约，不能绕过主轴直接拼接底层能力。

## 九、页面原型设计

本次原型采用简单 HTML 单页方式实现，主页链接如下：

[查看原型主页](实验报告1-原型主页.html)

该原型主页不是完整产品，而是把运行时核心功能压缩到一个静态页面中，用于直观展示系统结构与核心流程。

### 1. 原型主页：Runtime Core Overview

**设计目标：** 用一个页面展示运行时核心链路和关键能力单元。

**页面内容：**

1. 项目名称与一句话定位。
2. Runtime 主轴流程说明。
3. Provider / Binding 模块说明。
4. Agent / Capability 模块说明。
5. 统一输出模型说明。

**原型说明：**

- 页面顶部展示项目标题和实验定位。
- 中部使用分区卡片展示 `Runtime 主轴`、`Provider 与 Binding`、`Agent 与 Capability`、`统一输出` 四个核心区域。
- 页面底部放置一次请求的流转路径，强化“请求如何进入、如何执行、如何结束”的理解。

### 2. 功能区一：Provider 与 Binding

**设计目标：** 说明模型接入信息在运行时中的位置。

**页面内容：**

1. `custom provider` 配置项。
2. 提供商类型。
3. 探测结果。
4. 模型发现结果。
5. 最终 binding。

**原型说明：**

- 该区域强调“配置事实”和“运行时可消费 binding”之间的区别。
- 用流程箭头展示从保存配置到生成 binding 的过程。

### 3. 功能区二：Runtime 与 Agent 执行

**设计目标：** 说明请求进入系统后的执行链路。

**页面内容：**

1. Session。
2. Request Context。
3. Capability Set。
4. Koog `AIAgent`。
5. Event / Result / Failure。

**原型说明：**

- 用一条主流程线说明一次调用从输入到输出的变化。
- 明确 runtime 负责组织调用流，agent 负责执行语义。

### 4. 功能区三：Capability Integration

**设计目标：** 说明不同能力来源如何被统一接入。

**页面内容：**

1. Local / Custom Tools。
2. MCP。
3. Direct HTTP Internal API。
4. 统一 capability contract。

**原型说明：**

- 该区域不强调具体实现代码，而强调不同能力在接口层被统一收敛。
- 这样可以说明后续扩展能力时不需要重写上层调用流。

## 十、核心交互流程设计

### 1. Provider 创建与模型发现流程

```text
输入 provider 配置
-> 设置 Provider 名称、类型、baseUrl、apiKey
-> 触发保存与验证
-> 系统执行连接探测
-> 探测成功后自动发现模型
-> 生成模型列表与默认选择
-> 生成最终 binding
-> 回接 runtime 供后续请求消费
```

### 2. 会话执行流程

```text
输入请求内容
-> 选择 Provider 与模型 binding
-> 触发执行
-> Runtime 创建 session 与 request context
-> Runtime 解析 capability set
-> Koog agent 执行请求
-> 输出 RuntimeEvent / RuntimeResult / RuntimeFailure
```

### 3. 原型表达规则

1. Provider 类型变更后必须重新探测，不允许沿用旧模型列表。
2. 请求执行时必须显式展示状态，避免无法判断系统是否正在工作。
3. 错误展示需要对用户可读，不能只输出底层异常堆栈。
4. HTML 原型只负责说明结构和流程，不承担真实 runtime 逻辑。

## 十二、实验中遇到的问题及解决方案

### 1. 问题一：项目当前没有现成 Web 界面

**问题说明：** 当前仓库主线明确以 CLI 为第一入口，Web UI 只是后续可选表层，因此无法直接以“现成界面截图”的方式完成实验一。

**解决方案：** 将实验原型定位为“未来 Web 管理端的文本化原型设计”，以此满足实验对需求分析、信息架构和原型说明的要求，同时保证与项目现状一致。

### 2. 问题二：课程示例偏向 Axure 高保真原型

**问题说明：** 指导书示例主要以 Axure 为载体，而本次作业允许采用 Markdown 纯文本形式提交。

**解决方案：** 使用 Markdown + 页面说明 + 信息架构图 + 交互流程图的方式替代 Axure 页面，先完成设计表达；如果后续需要补强展示效果，可以再将本报告内容迁移到 HTML 或 Axure 中。

### 3. 问题三：技术文档不等于用户视角需求

**问题说明：** 项目原始文档偏架构和实现边界，直接提交会显得像技术说明书，而不是实验报告。

**解决方案：** 将技术概念重新映射为用户可感知的功能，如将 runtime 映射为“会话运行台”，将 provider/binding 映射为“模型配置与选择”，从而形成更符合课程要求的产品化表达。

## 十三、分析小结

通过本次实验，我完成了从真实软件项目中抽取需求、整理功能结构、设计页面层级并描述核心交互的全过程。虽然 `mulehang-agent` 当前尚未进入 Web UI 实现阶段，但通过对项目文档的分析，可以较清楚地定义未来界面的功能边界与交互方向。

本次实验也说明，原型设计并不等同于完整系统开发。对于项目早期阶段，更重要的是先把用户需求、系统结构、交互流程和模块关系表达清楚。只要这些内容足够准确，原型就已经发挥了沟通、验证和规划的作用。后续若需要继续完善，可在本报告基础上进一步补充 HTML 演示稿或 Axure 原型。
